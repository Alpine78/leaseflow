# LeaseFlow Production-Readiness Hardening Roadmap

## Purpose

This roadmap defines what would be required to move LeaseFlow from a validated
dev/portfolio MVP toward a production-like workload.

It is a planning document, not an implementation epic. The current MVP remains
valid for portfolio and dev demonstration use.

## Classification

MVP-blocking:

- None identified. The backend and infra MVP has already been validated for
  dev and portfolio use.

Production-blocking:

- Required before serious production or team-managed use.
- These items usually affect recoverability, operational control, security, or
  environment governance.

Optional hardening:

- Valuable controls that can be added later.
- These are not required for the current portfolio/demo goal.

## Terraform State And Environment Management

Classification: production-blocking

Current state:

- Dev now has an S3 remote-state bootstrap path with native Terraform S3
  lockfile locking.
- Existing local dev state can be migrated manually with
  `terraform init -backend-config=backend.hcl -migrate-state`.
- CI validates Terraform without AWS credentials.
- Dev stack creation and destroy are operator-driven.

Target posture:

- Remote encrypted Terraform state is consistently used for team-managed
  environments.
- Clear environment separation, at minimum dev and a production-like stage.
- Documented IAM access boundaries for state and lockfile operations.
- Documented state recovery process.

Cost impact: low to medium.

Well-Architected relevance:

- Operational Excellence: repeatable and reviewable changes.
- Reliability: recoverable infrastructure state.
- Security: encrypted state and controlled access.

SAA-C03 pitfall:

- Terraform files are not the same as Terraform state. Remote state reduces one
  operational risk, but production readiness still needs controlled access,
  recovery procedures, and environment separation.

## RDS Production Posture

Classification: production-blocking

Current state:

- Dev RDS is private and encrypted.
- `multi_az = false`
- `backup_retention_period = 1`
- `deletion_protection = false`
- `skip_final_snapshot = true`
- Stack destroy permanently removes dev data.

Target posture:

- Explicit backup retention aligned with recovery expectations.
- Deletion protection and final snapshot behavior reviewed for production-like use.
- Multi-AZ tradeoff reviewed against availability needs and cost.
- Restore objectives and evidence cadence defined before real tenant data exists.

Cost impact: medium to high.

Well-Architected relevance:

- Reliability: backups, recovery, and availability.
- Cost Optimization: Multi-AZ and retention increase cost.
- Operational Excellence: restore validation proves recoverability.

SAA-C03 pitfall:

- Backup retention is not the same as restore readiness. A workload is not
  recoverable until restore behavior has been validated.

## Secret Handling And Rotation

Classification: production-blocking

Current state:

- RDS password is generated by Terraform.
- Runtime password is stored in SSM Parameter Store `SecureString`.
- The password still exists in Terraform state because it is passed into RDS and
  SSM resources.

Target posture:

- Decide whether production should use AWS-managed master password handling,
  Secrets Manager, or another rotation-capable pattern.
- Define rotation procedure and emergency secret replacement procedure.
- Restrict who can read or change production secrets and Terraform state.

Cost impact: low to medium.

Well-Architected relevance:

- Security: secret lifecycle and least privilege.
- Operational Excellence: documented rotation and recovery.

SAA-C03 pitfall:

- Storing a value in SSM SecureString does not remove it from Terraform state if
  Terraform also manages the plaintext input.

## IAM Least-Privilege Review

Classification: production-blocking

Current state:

- Lambda IAM is scoped for logs, VPC networking, SSM parameter read, and KMS decrypt.
- VPC networking permissions still require broad EC2 network-interface actions.
- Scheduler has a dedicated invocation role.

Target posture:

- Review IAM policies with production threat model and deployment roles in mind.
- Separate Terraform execution permissions from runtime permissions.
- Document any wildcard IAM permissions that are required by AWS service behavior.

Cost impact: low.

Well-Architected relevance:

- Security: least privilege and blast-radius reduction.
- Operational Excellence: documented exceptions.

SAA-C03 pitfall:

- Least privilege is not only about Lambda runtime. Deployment identities and
  state access also need access boundaries.

## Alert Delivery And Escalation

Classification: production-blocking

Current state:

- Baseline CloudWatch alarms exist for Lambda errors, Lambda throttles, HTTP API
  5xx responses, and Scheduler target errors.
- Alarm state changes publish to an SNS topic.
- Dev can optionally add one email subscription, but delivery requires confirmation.

Target posture:

- Define who receives alerts, how they are acknowledged, and when escalation happens.
- Add OK/recovery notification policy only if it improves operations.
- Decide whether email, Slack, PagerDuty, or another channel is appropriate.
- Document alarm testing and expected CloudWatch evaluation delay.

Cost impact: low to medium.

Well-Architected relevance:

- Operational Excellence: actionable alerts and response process.
- Reliability: faster detection and recovery.

SAA-C03 pitfall:

- An SNS topic is only an alarm action target. It is not human notification
  delivery until a subscription exists and is confirmed.

## Smoke Tests Versus Scheduled Synthetics

Classification: optional hardening

Current state:

- Deployed smoke test runbook exists.
- One deployed smoke test evidence note exists.
- Local demo client can exercise the MVP path manually.
- No scheduled synthetic check exists.

Target posture:

- Decide whether production-like environments need scheduled synthetic checks.
- Keep synthetic checks tenant-safe and free of sensitive evidence.
- Separate demo tooling from continuous monitoring.

Cost impact: low to medium.

Well-Architected relevance:

- Operational Excellence: repeatable verification.
- Reliability: early detection of integration failures.

SAA-C03 pitfall:

- A runbook proves an operator can test the system. It does not prove the system
  is continuously monitored.

## DR Validation And Restore Objectives

Classification: production-blocking

Current state:

- Dev RDS restore validation runbook and evidence exist.
- Restore validation cadence is documented for MVP development.
- Private data-level validation path is documented, but not automated.

Target posture:

- Define RTO and RPO before production use.
- Align backup retention and restore validation cadence with those objectives.
- Add a private read-only data validation path if evidence needs to be executable.

Cost impact: medium.

Well-Architected relevance:

- Reliability: recovery objectives and tested restore process.
- Security: private validation without exposing restored data.

SAA-C03 pitfall:

- RDS automated backups provide restore capability, but application-level
  recovery still needs schema and data validation.

## Tenant Isolation Hardening

Classification: optional hardening for portfolio, production-blocking for real multi-tenant use

Current state:

- App-level tenant isolation is implemented.
- `tenant_id` comes from Cognito JWT claims.
- Tenant-scoped domain tables include `tenant_id`.
- Client-supplied `tenant_id` is not trusted.

Target posture:

- Evaluate PostgreSQL Row Level Security as defense in depth.
- Add cross-tenant negative tests for all future sensitive flows.
- Define operational controls for tenant membership changes.

Cost impact: low to medium.

Well-Architected relevance:

- Security: defense in depth and data isolation.
- Operational Excellence: repeatable validation of tenant boundaries.

SAA-C03 pitfall:

- Application-layer isolation can be valid, but real multi-tenant production
  systems often need additional controls and stronger operational processes.

## Cost Controls And Expected Cost Envelope

Classification: production-blocking for paid environments

Current state:

- Dev stack is cost-aware and should be destroyed when not needed.
- RDS is the primary expected dev cost driver.
- No NAT Gateway is used.

Target posture:

- Define monthly cost envelope per environment.
- Add budget alerts or cost review process before long-lived environments exist.
- Decide which production hardening items justify their ongoing cost.

Cost impact: low for governance, medium to high for HA and monitoring additions.

Well-Architected relevance:

- Cost Optimization: explicit cost tradeoffs and budget controls.
- Operational Excellence: cost visibility and ownership.

SAA-C03 pitfall:

- High availability and observability features are useful, but they are not free.
  Architecture decisions must match business recovery and cost requirements.

## Recommended Follow-Up Tickets

Create implementation tickets only when the project goal moves beyond portfolio
demo readiness.

Highest-value follow-ups:

1. Define production-like RDS protection baseline.
2. Add stricter Terraform state access boundaries if the project becomes
   team-managed.

Defer for now:

- Secret rotation implementation.
- Alert escalation integration.
- Scheduled synthetic checks.
- PostgreSQL Row Level Security.
- Multi-environment deployment workflow.
- Production cost monitoring.

## Decision Guidance

Choose portfolio/demo path when:

- the goal is to show cloud/backend architecture thinking
- short-lived dev deployments are enough
- cost control is more important than production realism

Choose production-hardening path when:

- multiple people need to collaborate on infrastructure
- real tenant data would be stored
- recovery expectations are explicit
- alerts must reach accountable humans
- uptime and restore objectives matter

Current recommendation:

- Keep LeaseFlow in portfolio/demo mode unless the project goal explicitly
  changes.
- Treat RDS protection and stricter Terraform state access boundaries as the
  first hardening work if the project moves toward team-managed or
  production-like operation.
